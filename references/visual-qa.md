# Visual QA

Review the rendered result in this order.

## Opening viewport

- Is there one obvious focal point within two seconds?
- Can a visitor identify the project category without reading a wall of text?
- Does the primary visual show the actual project or explain its system?
- Is owner-authored orientation visible without scrolling excessively?
- Are actions and badges subordinate to the identity and proof visual?
- Does anything look like a clickable control but lead nowhere?

## First third

- Does the sequence progress from identity to proof to next step?
- Is there enough negative space to separate those roles?
- Is repeated information removed or deferred?
- Does the first major section begin before the opening becomes self-indulgent?

## Whole-document rhythm

- Do headings reveal a coherent journey when scanned alone?
- Are long technical sections broken by meaningful examples or visuals?
- Are tables narrow enough for mobile?
- Are secondary details collapsed or linked instead of dominating?
- Does each visual have a job: identity, proof, explanation, navigation, or status?

## Theme and viewport

- Preview light and dark GitHub themes.
- Preview a narrow viewport around 375–430 CSS pixels.
- Check transparent logos against both backgrounds.
- Check diagram labels and screenshot annotations at rendered width.
- Confirm image crops do not hide the important interaction on narrow screens.

## Integrity

- Confirm every badge and action link resolves to the intended project resource.
- Confirm all local assets render from the README’s location.
- Confirm no secret, local filesystem path, owner placeholder, or temporary URL is visible.
- Confirm existing technical content was preserved unless its removal was in scope.
- Confirm generated visuals do not misrepresent the interface or architecture.

## Common rejection signals

Reject and revise when the README has:

- a badge wall before project identity;
- an enormous generic banner with no project evidence;
- centered long-form prose;
- several equally dominant screenshots;
- fake buttons or speculative links;
- decorative feature cards containing invented copy;
- broken dark-mode transparency;
- tiny diagram text;
- a wide table that requires horizontal scrolling for core information;
- the same visual formula the skill would apply to an unrelated repository.
